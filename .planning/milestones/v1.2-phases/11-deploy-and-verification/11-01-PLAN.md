  <action>
In `gbgolf/fetcher.py`, function `write_projections()` at line 94, replace:

```python
# Enable FK constraints for SQLite (no-op on PostgreSQL)
session.execute(text("PRAGMA foreign_keys = ON"))
```

With a dialect-conditional guard:

```python
# Enable FK constraints for SQLite (PostgreSQL always enforces FKs)
if session.bind.dialect.name == "sqlite":
    session.execute(text("PRAGMA foreign_keys = ON"))
```

This check uses `session.bind.dialect.name` which returns `"sqlite"` for SQLite connections and `"postgresql"` for PostgreSQL connections. PostgreSQL always enforces foreign key constraints, so the PRAGMA is unnecessary there and would raise a ProgrammingError.

Also update the comment on line 93 to remove "(no-op on PostgreSQL)" since it is NOT a no-op -- it would crash.

In `tests/test_fetcher.py`, the test functions `test_write_projections_inserts`, `test_write_projections_replaces_stale`, `test_write_projections_idempotent`, and `test_run_fetch_low_count_guard` all contain a line `db_session.execute(text("PRAGMA foreign_keys = ON"))` at the start of the test. These lines are fine because the tests use SQLite, but for consistency and clarity, add a comment above each one:

```python
# SQLite requires explicit FK enforcement (tests use in-memory SQLite)
db_session.execute(text("PRAGMA foreign_keys = ON"))
```

This makes it clear these are intentional SQLite-specific calls in the test context.

# Deploy GB Golf Optimizer to VPS
